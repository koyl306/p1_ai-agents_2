# README

## Опис завдання

Метою було провести рефакторинг функції валідації форми, а також проаналізувати зміни, запропоновані AI (ChatGPT 5.3 та GitHub Copilot на базі Claude Haiku 4.5), з точки зору читабельності, масштабованості та надійності.

Початковий код містив імперативну реалізацію з послідовними `if` перевірками. У процесі було створено три альтернативні варіанти реалізації та набір тестів для перевірки їх коректності.

---

## Що було знайдено

### 1. Проблеми читабельності (Script1.js)

* Використання масиву анонімних функцій ускладнює розуміння логіки.
* Неявні повернення через `&& 'error'` знижують прозорість коду.

### 2. Змішування логіки та даних (Script1.js)

* Валідаційні правила жорстко закодовані.
* Відсутня декларативність та гнучкість.

### 3. Потенційні помилки через `||` (Script2.js)

* Вирази можуть повертати неочікувані значення (не тільки `true` або рядок).

### 4. Відсутність явної перевірки `undefined` (Script2.js)

* Деякі правила покладаються на неявну поведінку JavaScript.

### 5. Надмірна абстракція (Script3.js)

* Винесення логіки у `runValidation` додає рівень непрямості.

### 6. Помилка з невизначеним `schema` (Script3.js)

* Код не працює без додаткового визначення.

### 7. Масштабованість

* В оригінальному коді складно додавати нові правила.

### 8. Втрата простоти

* Початковий варіант простіший і зрозуміліший для невеликих задач.

---

## Що було виправлено

* Винесено правила в окрему конфігурацію (`rules` / `schema`) для покращення масштабованості.
* Усунуто небезпечні конструкції з `||`, додано явне повернення `true` або повідомлення про помилку.
* Додано перевірки на `undefined` для коректної обробки даних.
* Виправлено помилку з відсутнім `schema`.
* Покращено архітектуру для повторного використання логіки.

---

## Що було відхилено

* Функціональний стиль (Script1) — через зниження читабельності.
* Надмірна абстракція (Script3) — як стилістичне рішення без значного виграшу.
* Спрощення оригінального коду — залишено як валідний варіант для малих проєктів.

---

## Тестування

Було створено 8 тестів (Jest):

### Для Script1:

* 1.1 — однакові тест-кейси для кількох реалізацій
* 1.2 — параметризовані тести (`test.each`)
* 1.3 — перевірка валідного вводу (очікується порожній масив)

### Для Script2:

* 2.1 — перевірка некоректного імені
* 2.2 — валідація email (з і без `@`)
* 2.3 — перевірка мінімальної довжини пароля

### Для Script3:

* 3.1 — перевірка невідповідності паролів
* 3.2 — валідація віку (граничні значення та невалідні випадки)

## Висновок

Рефакторинг покращив масштабованість і розширюваність коду за рахунок винесення правил у конфігурацію. Водночас було виявлено компроміс між гнучкістю та читабельністю. Найбільш збалансованим варіантом є підхід із декларативними правилами (Script2) після виправлення виявлених проблем.
